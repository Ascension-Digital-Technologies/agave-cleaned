# Source Integrity

This document explains how the cleaned repository tracks source-file integrity.

The main repository claim is intentionally narrow:

> This is an unofficial cleaned and restructured Agave workspace. Cleanup work should not intentionally change validator, runtime, SVM, program, networking, consensus, or cryptographic behavior.

## Manifest

The baseline source manifest lives at:

```text
docs/source-manifest.json
```

It records source-like files by normalized path, size, and SHA-256 digest. It is generated by:

```bash
make source-manifest
```

and verified by:

```bash
make verify-source-integrity
```

## What the manifest checks

The manifest covers source files with extensions commonly used by this workspace, including Rust, C, C++, headers, assembly, protobuf, and SBF-related source fixtures.

It intentionally does not treat Markdown documentation, repository metadata, GitHub workflow files, or developer scripts as validator source behavior.

## What the manifest does not prove by itself

The committed manifest proves that source files have not changed relative to the cleaned-repo baseline committed with this package. It does not, by itself, prove equivalence to an upstream Agave commit unless the upstream base is recorded and compared.

For upstream comparison, use:

```bash
scripts/dev/verify-upstream.sh /path/to/anza-agave
```

That command compares cleaned files against an upstream checkout using repository path translation rules.

## Release expectation

Before a public release, maintainers should:

1. record the upstream repository and commit in [`UPSTREAM.md`](../UPSTREAM.md),
2. regenerate `docs/source-manifest.json`,
3. run `make verify-source-integrity`,
4. optionally compare against a local upstream checkout with `scripts/dev/verify-upstream.sh`,
5. attach the manifest and release checksum files to the release artifacts.
